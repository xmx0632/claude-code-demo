## 干系人分类

### 按影响力和利益相关度分类

```
        高影响力
           |
    高关注  |  低关注
-----------+-----------
    高关注  |  低关注
           |
        低影响力
```

#### 高影响力 + 高利益相关 (关键干系人)
- **管理策略**: 重点管理，密切沟通
- **干系人**:
  - 产品负责人：决定产品方向和优先级
  - 技术负责人：决定技术方案和质量标准
  - 项目经理：控制项目进度和资源
- **沟通方法**:
  - 每日沟通
  - 定期会议
  - 及时汇报重大问题
  - 邀请参与关键决策

#### 高影响力 + 低利益相关
- **管理策略**: 让其满意，定期汇报
- **干系人**:
  - 无（暂无此类干系人）
- **沟通方法**:
  - 定期发送项目报告
  - 重要节点通知
  - 保持信息透明

#### 低影响力 + 高利益相关
- **管理策略**: 保持知情，充分利用
- **干系人**:
  - 开发工程师：具体实现功能，需要充分支持
  - 测试工程师：保证质量，需要及时沟通
- **沟通方法**:
  - 日常工作中密切配合
  - 及时提供所需信息
  - 听取技术建议

#### 低影响力 + 低利益相关
- **管理策略**: 最小精力，监控即可
- **干系人**:
  - 文档编写者：支持但不需要过度关注
  - 运维工程师：在需要时参与
- **沟通方法**:
  - 定期更新项目状态
  - 重要变更通知
  - 按需沟通

### 定期沟通

#### 每日站会
- **参与者**: 全体团队（产品负责人、技术负责人、开发工程师、测试工程师、项目经理）
- **时间**: 每天上午 10:00，15 分钟
- **形式**: 线上（Zoom/Teams）
- **内容**:
  - 昨天完成了什么
  - 今天计划做什么
  - 有什么阻碍或问题
- **产出**: 每日进度记录、问题跟踪

#### 周例会
- **参与者**: 全体团队
- **时间**: 每周五下午 3:00，1 小时
- **形式**: 线上/线下
- **内容**:
  - 本周进展回顾
  - 下周计划
  - 风险和问题讨论
  - 需要的支持
- **产出**: 周报、风险登记册更新

#### 需求评审会议
- **参与者**: 产品负责人、技术负责人、开发工程师、测试工程师
- **时间**: 每两周或按需，1-2 小时
- **形式**: 线下
- **内容**:
  - 新需求介绍
  - 需求澄清
  - 技术可行性讨论
  - 工作量估算
- **产出**: 用户故事确认、估算结果

#### 技术评审会议
- **参与者**: 技术负责人、开发工程师
- **时间**: 按需，1-2 小时
- **形式**: 线下
- **内容**:
  - 技术方案讨论
  - 架构设计评审
  - 技术难点讨论
- **产出**: 技术方案文档

#### 代码审查
- **参与者**: 技术负责人、开发工程师
- **时间**: 每日，持续进行
- **形式**: 线上（Git Merge Request）
- **内容**:
  - 代码质量检查
  - 最佳实践验证
  - 潜在问题识别
- **产出**: 代码审查意见

#### 迭代评审会议

  - 验收用户故事
- **产出**: 验收结果、反馈清单

## 干系人期望管理

### 期望记录

| 干系人 | 期望 | 优先级 | 状态 | 管理措施 |
|--------|------|--------|------|----------|
| 产品负责人 | 所有 P0 和 P1 功能按计划交付 | 高 | 讨论中 | 定期展示进度，管理范围变更 |
| 产品负责人 | 系统易用，用户体验良好 | 高 | 计划中 | 邀请参与设计和评审 |
| 技术负责人 | 代码符合规范，测试覆盖率高 | 高 | 计划中 | 代码审查，自动化测试 |
| 技术负责人 | 系统性能满足要求 | 中 | 计划中 | 性能测试，优化 |
| 开发工程师 | 需求清晰明确 | 高 | 持续中 | 需求评审，文档完善 |
| 开发工程师 | 技术支持及时 | 中 | 持续中 | 技术讨论，代码审查 |
| 测试工程师 | 测试时间充足 | 中 | 计划中 | 测试计划，时间分配 |
| 项目经理 | 项目按计划交付 | 高 | 跟踪中 | 每日站会，进度跟踪 |

### 期望管理策略

1. **明确沟通**
   - 清晰传达项目目标和范围
   - 定期更新项目进展
   - 透明分享问题和挑战
   - 使用项目看板（如 Jira/Trello）可视化进度

2. **设置合理期望**
   - 基于实际能力和资源设定目标
   - 考虑风险和不确定性（预留缓冲时间）
   - 留有余地（不过度承诺）
   - 使用估算区间而不是精确值

3. **及时反馈**
   - 快速响应干系人问题
   - 定期收集反馈
   - 调整策略和计划
   - 使用反馈改进流程

4. **管理范围蔓延**
   - 明确项目边界
   - 评估变更请求的影响
   - 使用优先级管理变更
   - 记录所有变更决策

## 干系人冲突管理

### 已识别冲突

| 冲突 | 涉及干系人 | 影响 | 解决方案 |
|------|-----------|------|----------|
| 功能完整性 vs 交付时间 | 产品负责人 vs 项目经理 | 高 | 使用优先级管理，优先 P0 功能，P1 功能根据时间调整 |
| 代码质量 vs 开发速度 | 技术负责人 vs 开发工程师 | 中 | 制定代码规范，使用自动化工具辅助，平衡质量和速度 |
| 测试时间 vs 开发时间 | 测试工程师 vs 开发工程师 | 中 | 提前制定测试计划，并行开发和测试，使用自动化测试 |
| 技术方案 vs 产品需求 | 技术负责人 vs 产品负责人 | 中 | 充分讨论技术可行性，寻求简化方案，优先 MVP |
| 文档详细度 vs 项目进度 | 文档编写者 vs 项目经理 | 低 | 优先核心文档，使用模板提高效率，持续更新 |

### 冲突解决流程

1. **识别冲突**
   - 收集冲突信息
   - 分析冲突原因
   - 评估影响（对项目进度、质量、干系人关系）
   - 记录到风险登记册

2. **促进沟通**
   - 组织协调会议
   - 让各方表达观点和关切
   - 寻找共同点和共同目标
   - 使用双赢思维

3. **寻找解决方案**
   - 头脑风暴多个方案
   - 评估每个方案的利弊
   - 选择最佳方案（或折中方案）
   - 考虑项目目标和约束

4. **跟踪和验证**
   - 记录解决方案和决策
   - 跟踪执行情况
   - 验证解决效果
   - 总结经验教训

### 冲突解决原则

1. **项目目标优先**: 冲突解决方案应有利于项目目标的实现
2. **双赢思维**: 寻求让各方都能接受的方案
3. **数据驱动**: 基于数据和事实而不是情绪做决策
4. **及时处理**: 不拖延冲突，尽早处理
5. **透明沟通**: 保持沟通透明，避免背后议论

### 反馈收集

#### 收集方式
1. **一对一访谈**
   - 与关键干系人定期一对一沟通
   - 深入了解关注点和期望
   - 收集不便公开讨论的反馈

2. **问卷调查**
   - 定期发送问卷给所有干系人
   - 收集满意度、建议等信息
   - 匿名方式鼓励真实反馈

3. **焦点小组**
   - 组织小组讨论会
   - 针对特定主题收集反馈
   - 促进干系人之间的交流

4. **演示后反馈**
   - 在迭代评审后收集反馈
   - 使用反馈表或在线工具
   - 及时记录和处理

#### 反馈处理流程
1. **记录所有反馈**: 使用统一的反馈记录表
2. **分类整理**: 按类型（功能、性能、体验等）和优先级分类
3. **评估优先级**: 评估影响和价值，确定处理优先级
4. **纳入需求或决策**: 高优先级反馈转化为需求或改进项
5. **反馈处理结果**: 向干系人反馈处理结果，形成闭环

## 总结

TodoList 项目作为 SDLC 框架的演示项目，干系人管理的关键在于：

1. **明确角色和职责**: 每个干系人都清楚自己的职责和期望
2. **有效沟通**: 建立定期的沟通机制，保持信息透明
3. **期望管理**: 合理设定和管理干系人期望，避免过度承诺
4. **冲突解决**: 及时识别和处理冲突，维护团队协作
5. **持续反馈**: 收集和处理干系人反馈，持续改进

通过有效的干系人管理，确保项目顺利完成，达到教育和演示的目标。
